# 生成 WebNN Vanilla JavaScript

Vanilla JavaScript 为 WebNN 应用程序提供最小的打包大小，使其成为对包大小敏感的部署的理想选择。然而，与 [Transformers.js](https://huggingface.co/docs/transformers.js/en/index) 或 [ONNX Runtime Web](https://onnxruntime.ai/) 等更高级别的框架相比，手动编写 WebNN 代码需要大量的开发工作。

WebNN 社区通过提供代码生成工具来解决这一挑战，这些工具可以自动将现有的机器学习模型转换为 WebNN JavaScript 的优化，从而提供性能和最大的吞吐量。

## 打包大小与开发效率

在构建基于 Web 的 ML 应用程序时，开发人员面临一个关键的权衡：

- **高级库**（ [Transformers.js](https://huggingface.co/docs/transformers.js/en/index) 、 [ONNX Runtime Web](https://onnxruntime.ai/) ）提供快速开发，但包大小更大
- **Vanilla WebNN JavaScript** 提供最小的占用空间，但需要大量的手动编码
- **生成的WebNN代码**结合了两者的优点：自动化开发和优化的包大小

| Feature | ONNX2WebNN | Model2WebNN | WebNNUtils/OnnxConverter |
|---------|:----------:|:-------------------:|:------------------------:|
| **Interface** | Command-line (CLI) | Web application, CLI, API | Python toolkit |
| **Language** | Python | TypeScript | Python |
| **ONNX Support** | ✅ Yes | ✅ Preferred | ✅ Yes |
| **TensorFlow Lite Support** | ❌ Not supported | ✅ Yes | ❌ Not supported |
| **Other Formats Support** | ❌ Not supported | ❌ Not supported | ❌ Not supported |
| **Privacy** | Client-side only | Client-side only | Client-side only |

### [ONNX2WebNN](https://github.com/huningxin/onnx2webnn)
- **最适合**：自动化工作流程、CI/CD管道、批处理
- **优势**：成熟的Python生态系统、命令行自动化
- **示例**：在开发管道中转换单个模型和多个模型

### [Model2WebNN](https://ibelem.github.io/model2webnn/)
- **最适合**：饮料开发、快速原型设计、TS/JS集成
- **优势**：Web UI、CLI和库API支持、TFLite支持、基于浏览器的隐私
- **示例**：直接在浏览器中或通过 CLI 转换具有完整数据隐私的单个模型

### [WebNNUtils/OnnxConverter](https://github.com/MicrosoftEdge/WebNNUtils/tree/main/OnnxConverter)
- **最适合**：部署生产、复杂模型优化
- **优势**：高级图形分区、模型重新排序、Microsoft Edge团队支持
- **示例**：需要通过CPU/NPU加速优化WebNN部署的企业应用程序

## 入门

1. **评估您的要求**：打包大小、隐私需求、开发工作流程、性能要求
2. **选择您的工具**：CLI自动化(ONNX2WebNN)、高级优化(WebNNUtils/OnnxConverter)或浏览器界面(Model2WebNN)
3. **准备模型**：确保Web部署的ONNX兼容性和优化
4. **生成和测试**：转换模型并验证目标环境中的性能
5. **部署和监控**：集成生成的代码和监控实际性能

WebNN 代码生成生态系统提供了灵活的选项，用于在 Web 应用程序中部署优化的 ML 推理，同时保持普通 JavaScript 的性能优势。
