# WebNN 测试

本指南涵盖了可用于跨不同浏览器和设备验证 WebNN 实现的测试方法和工具。

## 先决条件

在测试 WebNN 功能之前，请确保您已按照目标后端（例如 WebNN Windows ML / OpenVINO EP GPU）的 [WebNN Installation](./installation.mdx) 指南正确配置浏览器。

## 测试计划

WebNN测试包含多种验证方法来覆盖各种[test suites](#webnn-test-suites)，以确保正确性、性能、安全性和稳定性。

### 测试基础设施

- **Chromium 机器人**：持续集成测试涵盖单元测试和 WPT 一致性测试
- Microsoft Edge 团队测试设施
- [WebNN Automation Tests](https://github.com/webgfx/webnn-test) ，涵盖 WPT、示例和演示
- W3C WebML 社区测试设施
- [WebNN Samples Test Framework](https://github.com/webmachinelearning/webnn-samples-test-framework) ，涵盖 WebNN 示例和演示
- 其他硬件供应商测试设施

### 后端

WebNN 针对后端实现的多个版本进行了验证：

- **公开版本**：稳定的 Windows ML、执行提供程序以及 GPU、NPU 驱动程序
- IHV EP 认证测试套件
- **预发布**

### 测试设备

WebNN 确保在所有主要硬件配置上进行测试，并及时支持最新设备。

### 测试节奏

除了对所有代码更改进行持续集成测试之外，WebNN 还遵循预定的测试策略：

- **每晚**：WPT 和演示自动化测试
- **每日**：性能和模型测试
- **连续模糊测试**：模糊目标由构建机器人自动发现，使用不同的模糊引擎构建，然后上传到分布式 ClusterFuzz 系统以大规模运行
- **按需**：在主要版本之前进行页面堆测试以进行调试

## WebNN 测试套件

### Web 平台测试 (WPT)

WebNN API 遵循 W3C 标准并包含全面的测试套件以确保跨浏览器兼容性。

| Resource | Link |

#### WebNN Web 平台测试/一致性测试套件

[WebNN Conformance Test Suite (CTS)](https://github.com/web-platform-tests/wpt/tree/master/webnn) 是 [web-platform-tests](https://github.com/web-platform-tests/) 项目的一部分，该项目是一个跨浏览器测试套件，可确保 Web 平台跨所有主要浏览器的兼容性。

这些测试为以下方面提供了信心：

- **浏览器实现者** 他们的 WebNN 实现与其他浏览器兼容
- **Web 开发人员** 他们可以依赖跨浏览器和设备的一致 WebNN 行为

##### Web IDL 测试

**文件：** `idlharness.https.any.js`

验证 WebNN API 是否正确实现了 Web IDL 规范中定义的接口。该测试使用位于 `../interfaces/webnn.idl` 的规范 Web IDL 的每日更新快照。

##### 验证测试

**地点：** `validation_tests/`

这些测试验证 WebNN 方法根据规范算法正确验证输入参数。它们确保无效的输入被正确拒绝。

**示例：** `validation_tests/matmul.https.any.js` 确认：
- 当输入张量秩小于 2 时抛出异常
- 输入张量形状必须是可广播的
- 满足其他参数验证要求

##### 一致性测试

**地点：** `conformance_tests/`

这些测试验证 WebNN 运算符产生正确的计算结果。

**示例：** `conformance_tests/matmul.https.any.js` 验证 N 维矩阵乘法可产生数学上准确的输出。

#### 测试执行上下文

##### 设备变化
WebNN 支持多种计算设备（CPU、GPU、NPU）。测试使用 `META: variant=?...` 在不同的设备类型上自动运行相同的测试。

##### 全球背景
WebNN 接口可通过 `navigator.ml` 在 Window 和 Worker 上下文中使用。测试使用：
- `.any.js` 约定
- `META: global=window,dedicatedworker`

这允许相同的测试在多个 JavaScript 执行上下文中运行。

**注意：** 大多数特定于运算符的测试仅指定 `global=window` 以减少计算开销，因为各个运算符的上下文之间的行为差​​异不太可能。

### Chromium 中的模糊测试

模糊测试是一种自动化软件测试技术，它提供无效的、意外的或随机的数据作为程序的输入。它是发现代码中的安全漏洞和稳定性问题的最有效方法之一。

在 WebNN 的背景下，模糊测试通过生成传统测试用例可能无法覆盖的各种输入组合来帮助识别边缘情况和潜在的崩溃。

### PageHeap 下的测试

> <InfoIcon /> 注意：全页堆验证由于内存消耗较高，会导致系统内存不足。应在开发和测试期间有选择地使用它，而不是在生产环境中。

####什么是页堆？

PageHeap 是一项 Windows 诊断功能，可帮助检测内存访问违规。它在每次分配的边界处保留内存，以检测访问分配区域之外的内存的尝试。这可以捕获常见的内存错误，例如：

- 缓冲区溢出
- 释放后使用错误
- 越界内存访问

#### 页堆模式

GFlags 提供了两种 PageHeap 验证模式：

- **标准堆验证** 在每个堆分配的末尾写入填充模式，并在释放分配时检查这些模式。此模式对性能影响最小。
- **全页堆验证** 在每次分配的末尾放置一个不可访问的保护页，导致程序在访问超出分配范围的内存时立即停止。这提供了更强的保证，但使用了更多的内存（每次分配一个整页）。

#### 为什么要在 PageHeap 下测试 WebNN？

在 PageHeap 下测试 WebNN 有几个好处：

- **早期检测**：内存违规在发生时立即被捕获，使错误更容易诊断和修复。
- **安全保证**：帮助在生产之前识别与内存处理相关的潜在安全漏洞。
- **全面覆盖**：与模糊测试结合使用时，PageHeap 可确保自动生成的测试输入不会触发隐藏的内存安全问题。
